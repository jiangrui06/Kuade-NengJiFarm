## 一、概述与设计原则

### 1.1 背景

农场管理后台需要对**菜品订单**和**产品订单**提供后台退款能力：
- **菜品订单**（dish order）：现场点餐场景，支付后如遇菜品缺货/客户退款，管理员从后台发起退款
- **产品订单**（product order）：农产品商城场景，涉及发货/运输/自提等履约环节，退款流程更复杂，包含申请退款、处理退款、驳回退款三个阶段

### 1.2 统一设计原则

| 原则 | 说明 |
|------|------|
| **幂等性** | 同一笔订单重复发起退款不会重复扣款，后端需校验当前状态是否允许退款 |
| **状态校验** | 仅特定状态下的订单允许退款，已退款/已取消/未支付的订单拒绝退款 |
| **退款记录** | 每次退款操作需写入退款记录表，记录退款金额、原因、操作人、时间等 |
| **原子操作** | 退款操作需在同一个事务中完成：更新订单状态 + 写入退款记录 + 调用支付退款 |
| **可追溯** | 退款记录需包含完整审计信息：退款ID、退款编号、金额、原因、状态、图片证明、处理备注 |

### 1.3 通用响应格式

```json
{
  "code": 200,
  "message": "退款成功",
  "data": {
    "refundId": "RF20260525120001",
    "refundNo": "RF20260525120001",
    "refundAmount": 158.00,
    "status": "退款完成"
  }
}
```

## 二、菜品订单退款接口

### 2.1 接口信息

```
POST /api/dish/order/refund
Content-Type: application/json
Authorization: Bearer {token}
```

### 2.2 请求体

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| orderNo | string | **是** | 菜品订单号（订单唯一标识） |
| refundReason | string | 否 | 退款原因（选填，前端可留空） |

```json
{
  "orderNo": "D20260525120001",
  "refundReason": "菜品缺货"
}
```

### 2.3 业务规则

| 校验项 | 规则 |
|--------|------|
| 订单号 | 不能为空，必须在菜品订单表中存在 |
| 支付状态 | 仅 `已支付` 状态允许退款 |
| 订单状态 | `待支付` / `已取消` 状态拒绝退款，`备餐中` / `已完成` 状态视业务可退 |
| 后厨状态 | `已出餐` 状态建议弹窗二次确认是否允许退款 |
| 重复退款 | 已退款的订单返回错误"该订单已退款" |

### 2.4 后端处理流程

```
1. 根据 orderNo 查询菜品订单
2. 校验订单存在性 → 不存在返回 404
3. 校验支付状态 → 非已支付返回错误
4. 校验是否已退款 → 已退款返回错误
5. 调用微信支付退款接口（原路退回）
6. 写入退款记录表（refund_record）
7. 更新订单状态：
   - paymentStatus → "已退款"
   - orderStatus → "已取消"
   - kitchenStatus → "已取消"
8. 返回退款结果
```

### 2.5 成功响应

```json
{
  "code": 200,
  "message": "退款成功",
  "data": {
    "refundId": 1001,
    "refundNo": "RF20260525120001",
    "orderNo": "D20260525120001",
    "refundAmount": 158.00,
    "refundReason": "菜品缺货",
    "status": "退款完成",
    "refundTime": "2026-05-25 12:00:00"
  }
}
```

### 2.6 错误响应示例

```json
// 订单不存在
{ "code": 404, "message": "菜品订单不存在" }

// 已退款
{ "code": 400, "message": "该订单已退款，不能重复操作" }

// 支付状态不允许
{ "code": 400, "message": "当前订单支付状态为待支付，不允许退款" }

// 微信退款失败
{ "code": 500, "message": "微信退款接口调用失败，请稍后重试" }
```

## 三、产品订单退款接口

产品订单退款分**三个场景**，对应不同的接口调用：

| 场景 | 接口 | 方法 | 触发条件 |
|------|------|------|----------|
| 申请退款 | `/api/product/order/refund` | POST | 待发货/运输中 + 已支付 |
| 处理退款 | `/api/product/order/updateStatus` | PUT | 退款中 + 已支付 |
| 驳回退款 | `/api/product/order/updateStatus` | PUT | 退款中 + 已支付 |

### 3.1 申请退款（独立退款接口）

```
POST /api/product/order/refund
Content-Type: application/json
Authorization: Bearer {token}
```

#### 请求体

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| orderId | int | **是** | 产品订单主键ID（数字型，非订单号） |
| refundReason | string | 否 | 退款原因（选填） |

```json
{
  "orderId": 12345,
  "refundReason": "商品缺货，无法发货"
}
```

> **重要**：此处 `orderId` 是数据库主键（数字型），不是订单号 `orderNo`。前端通过 `getProductRefundOrderId()` 方法从订单对象中提取。

#### 业务规则

| 校验项 | 规则 |
|--------|------|
| orderId | 不能为空，必须在产品订单表中存在 |
| 订单状态 | 仅 `待发货` 或 `运输中` 状态允许申请退款 |
| 支付状态 | 仅 `已支付` 状态允许 |
| 订单类型 | 认购订单（orderCategory=subscription 或 orderId 以 S 开头）不允许退款 |
| 重复申请 | 已存在退款中/已退款的订单返回错误 |

#### 处理流程

```
1. 根据 orderId 查询产品订单
2. 校验订单存在性
3. 校验订单状态（待发货/运输中）
4. 校验支付状态（已支付）
5. 校验是否排除认购订单
6. 调用支付退款（原路退回）
7. 写入退款记录表
8. 更新订单状态：orderStatus → "已退款"，paymentStatus → "已退款"
9. 返回退款结果
```

#### 成功响应

```json
{
  "code": 200,
  "message": "退款成功",
  "data": {
    "refundId": 2001,
    "refundNo": "RF20260525120002",
    "orderId": 12345,
    "refundAmount": 299.00,
    "refundReason": "商品缺货，无法发货",
    "status": "退款完成",
    "refundTime": "2026-05-25 12:05:00"
  }
}
```

### 3.2 处理退款（状态更新接口）

```
PUT /api/product/order/updateStatus
Content-Type: application/json
Authorization: Bearer {token}
```

#### 请求体

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| orderNo | string | **是** | 产品订单号 |
| action | string | **是** | 固定值 `"refund-process"` |

```json
{
  "orderNo": "P20260525120001",
  "action": "refund-process"
}
```

#### 业务规则

| 校验项 | 规则 |
|--------|------|
| orderNo | 必须存在 |
| action | 必须为 `"refund-process"` |
| 订单状态 | 仅 `退款中` 状态允许处理 |
| 支付状态 | 仅 `已支付` 状态允许（非已退款） |

#### 处理流程

```
1. 查询订单，校验状态
2. 执行退款操作（如尚未调用微信退款）
3. 更新退款记录状态为"退款完成"
4. 更新订单状态：orderStatus → "已退款"，paymentStatus → "已退款"
5. 返回结果
```

---

### 3.3 驳回退款（状态更新接口）

```
PUT /api/product/order/updateStatus
Content-Type: application/json
Authorization: Bearer {token}
```

#### 请求体

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| orderNo | string | **是** | 产品订单号 |
| action | string | **是** | 固定值 `"refund-reject"` |
| adminReply | string | 否 | 驳回说明（展示给用户） |
| processNote | string | 否 | 处理备注（内部使用） |

```json
{
  "orderNo": "P20260525120001",
  "action": "refund-reject",
  "adminReply": "商品已发货，不符合退款条件",
  "processNote": "物流单号SF123456已揽件"
}
```

#### 业务规则

| 校验项 | 规则 |
|--------|------|
| orderNo | 必须存在 |
| action | 必须为 `"refund-reject"` |
| 订单状态 | 仅 `退款中` 状态允许驳回 |
| adminReply | 建议填写但不强制，驳回时应给用户明确原因 |

#### 处理流程

```
1. 查询订单，校验状态
2. 更新退款记录状态为"已驳回"
3. 写入 adminReply 和 processNote 到退款记录
4. 恢复订单状态：orderStatus → 回退到驳回前状态（如"待发货"）
5. 返回结果
```

## 四、退款状态流转

### 4.1 菜品订单退款状态机

```
待支付 ──(支付)──► 备餐中 ──(退款)──► 已取消(paymentStatus=已退款)
                      │
                      └──(出餐)──► 已完成
```

菜品订单退款是**一键式**：管理员在详情页点"退款"，直接完成退款。

### 4.2 产品订单退款状态机

```
待支付 ──(支付)──► 待发货 ──(申请退款)──► 已退款
                      │                      ▲
                      │                      │
                      └──(发货)──► 运输中 ──(申请退款)──┘
                                      │
                        ┌─────────────┘
                        ▼
                    退款中 ──(处理退款)──► 已退款
                      │
                      └──(驳回退款)──► 待发货（恢复）
```

产品订单退款是**两阶段**（特殊情况）：
1. **退款中**：用户先申请退款，订单进入"退款中"状态
2. **管理员审核**：可以处理退款（同意）或驳回退款（拒绝）

同时保留了管理员**直接申请退款**的能力（一键退款，跳过审核）。

### 4.3 状态映射表

| 订单类型 | 可退款状态 | 退款后状态 | 不可退款状态 |
|----------|-----------|-----------|-------------|
| 菜品订单 | 备餐中、已完成 | 已取消 | 待支付、已取消、已退款 |
| 产品订单 | 待发货、运输中（申请退款）| 已退款 | 待支付、已完成、已退款 |
| 产品订单 | 退款中（处理退款）| 已退款 | — |
| 产品订单 | 退款中（驳回退款）| 恢复原状态 | — |

## 五、前端调用速查

### 5.1 菜品订单退款

```javascript
// 方式一：通过 FarmAPI 封装（推荐）
const data = await FarmAPI.dishOrder.refund({
  orderNo: 'D20260525120001',
  refundReason: '菜品缺货'  // 可选
});

if (FarmAPI.isSuccessResponse(data)) {
  console.log('退款成功', data.data);
} else {
  console.error('退款失败', FarmAPI.getErrorMessage(data));
}

// 方式二：直接 fetch
const response = await fetch(FarmAPI.url('/api/dish/order/refund'), {
  method: 'POST',
  headers: Auth.getAuthHeaders({ 'Content-Type': 'application/json' }),
  body: JSON.stringify({
    orderNo: 'D20260525120001',
    refundReason: '菜品缺货'
  })
});
```

### 5.2 产品订单 — 申请退款

```javascript
// 注意：产品订单退款接口未封装在 FarmAPI.productOrder 中
// 前端直接调用 fetch

const orderId = getProductRefundOrderId(order);  // 提取数字型主键
const response = await fetch(FarmAPI.url('/api/product/order/refund'), {
  method: 'POST',
  headers: Auth.getAuthHeaders({ 'Content-Type': 'application/json' }),
  body: JSON.stringify({
    orderId: orderId,
    refundReason: '商品缺货'
  })

const result = await FarmAPI.parseResponse(response);
if (FarmAPI.isSuccessResponse(result)) {
  // 刷新列表
```

## 六、错误码与异常处理

### 6.1 HTTP 状态码

| 状态码 | 含义 |
|--------|------|
| 200 | 请求成功（业务结果看 code/message） |
| 400 | 参数校验失败或业务不允许 |
| 401 | 未登录/token 过期 |
| 403 | 无操作权限 |
| 404 | 订单不存在 |
| 500 | 服务器内部错误/微信退款失败 |

### 6.2 业务错误消息

| 错误消息 | 触发条件 |
|----------|----------|
| `订单号不能为空` | orderNo 未传 |
| `订单ID不能为空` | orderId 未传（产品订单） |
| `菜品订单不存在` | orderNo 查不到菜品订单 |
| `产品订单不存在` | orderId 查不到产品订单 |
| `该订单已退款，不能重复操作` | 订单已处于退款完成状态 |
| `当前订单支付状态为待支付，不允许退款` | 支付状态不是已支付 |
| `当前订单状态不允许退款` | 订单状态不在可退款白名单 |
| `认购订单不支持退款` | 订单类型为认购（subscription） |
| `微信退款接口调用失败` | 支付网关退款异常 |
| `退款记录写入失败` | 数据库写入异常 |

### 6.3 前端异常处理策略

```javascript
async function safeRefund(apiCall) {
  try {
    const result = await apiCall();
    if (!FarmAPI.isSuccessResponse(result)) {
      const msg = FarmAPI.getErrorMessage(result, '退款失败');
      return { success: false, message: msg };
    }
    return { success: true, data: result.data };
  } catch (error) {
    if (error.message.includes('Failed to fetch')) {
      return { success: false, message: '网络连接失败，请检查网络后重试' };
    }
    return { success: false, message: error.message || '退款请求异常' };
  }
}
```

## 七、数据库退款记录表设计参考

### 7.1 refund_record 表

| 字段 | 类型 | 说明 |
|------|------|------|
| id | bigint (PK, AUTO) | 主键 |
| refund_no | varchar(32) | 退款编号（如 RF20260525120001） |
| order_no | varchar(32) | 关联订单号 |
| order_type | varchar(20) | 订单类型：dish / product |
| refund_amount | decimal(10,2) | 退款金额 |
| refund_reason | varchar(500) | 退款原因 |
| status | varchar(20) | 退款状态：退款中 / 退款完成 / 已驳回 |
| admin_reply | varchar(500) | 管理员回复（驳回时填写） |
| process_note | varchar(500) | 处理备注 |
| refund_proof_images | text | 退款证明图片（JSON数组） |
| operator_id | int | 操作人ID |
| operator_name | varchar(50) | 操作人姓名 |
| create_time | datetime | 创建时间 |
| update_time | datetime | 更新时间 |
| wx_refund_id | varchar(64) | 微信退款单号 |

### 7.2 订单表退款相关字段

菜品订单表和产品订单表需包含：

| 字段 | 类型 | 说明 |
|------|------|------|
| payment_status | varchar(20) | 支付状态（待支付/已支付/已退款） |
| order_status | varchar(20) | 订单状态 |
| refund_id | bigint | 关联退款记录ID |
