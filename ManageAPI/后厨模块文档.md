# 后厨管理模块文档

## 一、模块概述

后厨管理模块负责**扫码点餐订单的出餐流程**，核心数据表为 `dish_orders`（菜品订单）和 `dish_order_details`（菜品订单明细）。

- **路由前缀**: `/api/Kitchen`
- **认证方式**: JWT Token（Header: `Authorization: Bearer <token>`）
- **权限要求**: 仅 RoleId=4 的后厨角色可登录

---

## 二、数据库表关系

```
dish_orders (订单)
  ├── order_id          → 主键
  ├── order_no          → 订单编号
  ├── order_status_id   → 订单状态 (1/2/3/4)
  ├── dining_table_id   → 关联桌台
  ├── total_amount      → 订单总金额
  ├── user_id           → 下单用户
  ├── create_time       → 创建时间
  └── remark            → 备注

dish_order_details (订单明细)
  ├── dish_order_details_id → 主键
  ├── dish_order_id     → 关联 dish_orders.order_id
  ├── dish_id           → 关联菜品
  ├── status_id         → 出餐状态 (1/2/3)
  ├── unit_price        → 单价
  ├── quantity          → 数量
  └── subtotal_amount   → 小计金额

dining_tables (桌台)
  ├── dining_table_id   → 主键
  └── table_no          → 桌号
```

## 三、状态码定义

### 3.1 订单状态 (`dish_orders.order_status_id`)

| 状态码 | 含义 | 说明 |
|--------|------|------|
| 1 | 待付款 | 后厨不显示 |
| 2 | 待出餐 | 已付款，等待后厨出餐 |
| 3 | 已完成 | 全部菜品已出餐 |
| 4 | 已取消 | 订单已取消 |

### 3.2 菜品明细状态 (`dish_order_details.status_id`)

| 状态码 | 含义 | 说明 |
|--------|------|------|
| 1 | 待出餐 | 等待后厨制作 |
| 2 | 已出餐 | 已完成出餐 |
| 3 | 已取消 | 该菜品被取消 |

### 3.3 状态流转规则

```
订单:  1(待付款) → 2(待出餐) → 3(已完成)
                     ↘ 4(已取消)

菜品:  1(待出餐) → 2(已出餐)
         ↘ 3(已取消)
```

**约束**:
- 已出餐的菜品（status_id=2）不可取消
- 订单所有菜品都变为"已出餐"时，订单自动变为"已完成"(3)
- 单个菜品标记出餐时，订单状态从 2→3（仅当订单状态不为 3 时才更新）

## 四、API 接口

### 4.1 后厨登录

```
POST /api/Kitchen/login
```

**请求体**:
```json
{
    "phoneNumber": "13800138000",
    "password": "123456"
}
```

**成功响应**:
```json
{
    "code": 200,
    "data": {
        "token": "eyJhbGciOi...",
        "user_id": 42,
        "user_name": "张三",
        "phone_number": "13800138000"
    }
}
```

**错误场景**:
- 手机号未注册 → "该手机号未注册"
- 密码错误 → "账号或密码错误"
- 角色权限不足 → "您的账号没有访问后厨系统的权限"

---

### 4.2 获取订单列表（按出餐状态筛选）

```
GET /api/Kitchen/order/list?type={type}
```

| 参数 | 类型 | 必填 | 说明 |
|------|------|------|------|
| type | int | 是 | 2=待出餐订单, 3=已完成订单 |

**响应示例**:
```json
{
    "code": 200,
    "data": [
        {
            "id": 1001,
            "no": "DO202605120001",
            "time": "2026-05-12 14:30:00",
            "table": "A05",
            "total": 128.50,
            "items": [
                {
                    "dishOrderDetailsId": 2001,
                    "name": "红烧排骨",
                    "status": 1,
                    "quantity": 2,
                    "price": 45.00
                }
            ]
        }
    ]
}
```

---

### 4.3 获取订单详情

```
GET /api/Kitchen/order/detail?orderId={orderId}
```

返回订单完整信息，含所有菜品的出餐状态名称。

---

### 4.4 标记菜品已出餐（核心接口）

```
POST /api/Kitchen/dish/finish
```

**请求体**:
```json
{
    "dishOrderDetailsId": 2001
}
```

**逻辑说明**:
1. 根据 `dishOrderDetailsId` 查询菜品明细
2. 将该明细所属订单的 `order_status_id` 更新为 **3（已完成）**
3. 幂等保证：如果订单状态已经是 3，则不再更新

**注意**: 此接口标记的是**整个订单**完成，而非单个菜品。如需按菜品粒度出餐，前端应调用此接口后自行管理。

---

### 4.5 取消菜品

```
POST /api/Kitchen/dish/cancel
```

**请求体**:
```json
{
    "dishOrderDetailsId": 2001
}
```

**约束**: 已出餐的菜品（status_id=2）不可取消。

**成功响应**:
```json
{
    "code": 200,
    "data": {
        "dishOrderDetailsId": 2001,
        "status": 3
    }
}
```

---

### 4.6 今日统计数据

```
GET /api/Kitchen/today-statistics
```

**响应示例**:
```json
{
    "code": 200,
    "data": {
        "todayTotalAmount": 3850.00,
        "todayTotalOrder": 32,
        "todayFinishedOrder": 28,
        "todayPendingDish": 12,
        "todayFinishedDish": 45
    }
}
```

| 字段 | 说明 |
|------|------|
| todayTotalAmount | 今日营业额（已出餐菜品小计汇总）|
| todayTotalOrder | 今日订单总数 |
| todayFinishedOrder | 已完成订单数（所有菜品均为已出餐）|
| todayPendingDish | 待出餐菜品数（status_id=1）|
| todayFinishedDish | 已出餐菜品数（status_id=2）|

---

### 4.7 退出登录

```
POST /api/Kitchen/logout
```

## 五、认证流程

```
1. 客户端调用 POST /api/Kitchen/login
       ↓
2. KitchenService.LoginAsync() 验证手机号+密码
       ↓
3. 校验 RoleId 是否 = 4（后厨角色）
       ↓
4. JwtHelper.GenerateToken() 生成 Token
       ↓
5. 后续请求在 Header 中携带: Authorization: Bearer <token>
```
