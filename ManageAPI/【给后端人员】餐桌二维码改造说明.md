# 餐桌二维码改造说明 —— 给后端人员

## 一、背景

目前管理后台（`web_management`）的餐桌模块已对接了后端的以下几个接口：

| 接口 | 方法 | 路径 |
|---|---|---|
| 新增餐桌 | POST | `/api/table/add` |
| 编辑餐桌 | POST | `/api/table/edit` |
| 餐桌列表 | GET | `/api/table/list` |
| 餐桌详情 | GET | `/api/table/detail/{id}` |
| 删除餐桌 | POST | `/api/table/delete` |
| 更新状态 | POST | `/api/table/status` |

当前前端新增/编辑餐桌时，后端返回的数据中有一个 `qrCodeUrl` 字段，前端会优先使用它来展示和打印二维码。**但这个字段目前是空的或者指向了错误的内容**，导致扫码无效。

---

## 二、需要后端做的事

### 2.1 新增/编辑餐桌时生成正确的二维码图片

后端在 `POST /api/table/add` 和 `POST /api/table/edit` 接口中，需要在处理完餐桌数据后，额外做以下操作：

**步骤1：生成二维码内容**

二维码中应该编码的是微信小程序跳转链接，格式如下（固定值，不可改变）：

```
weixin://dl/business/?appid=wx986e22f241e13ba2&path=subpkg/order/order&query=tableId={tableId}&secret=^mFIT!xzJ@j55QN%R^4yZ0vx
```

参数说明：

| 参数 | 值 | 说明 |
|---|---|---|
| `appid` | `wx986e22f241e13ba2` | 微信小程序 AppID，固定字符串 |
| `path` | `subpkg/order/order` | 小程序点餐页面路径，固定字符串 |
| `tableId` | 实际桌台 ID（如 1、2、3...） | 从餐桌数据中获取 |
| `secret` | `^mFIT!xzJ@j55QN%R^4yZ0vx` | 固定密钥，小程序端会用此值校验身份 |

**步骤2：生成二维码图片并保存**

- 使用二维码生成库（如 `qrcode`）将上述链接转为二维码图片
- 图片保存到服务器路径：`/images/qrcode/table_{tableId}.png`
- 例如：`/images/qrcode/table_1.png`、`/images/qrcode/table_2.png`

**步骤3：响应中返回 qrCodeUrl**

接口响应中必须返回 `qrCodeUrl` 字段，值为该图片的可访问 URL：

```json
{
  "code": 200,
  "message": "新增成功",
  "data": {
    "id": "xxx",
    "tableno": "1号桌",
    "capacity": 4,
    "status": 1,
    "qrCodeUrl": "https://你的域名/images/qrcode/table_1.png"
  }
}
```

### 2.2 配置静态资源路由

确保 `/images/qrcode/` 目录下的图片可以通过公网访问，否则前端展示和打印时图片无法加载。

### 2.3 餐桌详情接口也返回 qrCodeUrl

`GET /api/table/detail/{id}` 接口同样需要返回该桌台的 `qrCodeUrl`，因为管理后台的"查看明细"弹窗中需要展示二维码图片。

---

## 三、数据库字段建议

如果目前数据库没有存储二维码路径的字段，建议在餐桌表中增加一个字段：

| 字段名 | 类型 | 说明 |
|---|---|---|
| `qr_code_url` | varchar(512) | 二维码图片的 URL 或路径 |

---

## 四、验证方式

1. 调用 `POST /api/table/add` 新增一张餐桌
2. 检查响应中是否返回了 `qrCodeUrl` 字段，且 URL 可以正常访问
3. 用微信扫一扫扫描该图片，应能跳转到小程序的点餐页面，并携带正确的 `tableId` 和 `secret` 参数
4. 小程序端校验 `secret` 通过后，即可完成桌台绑定
